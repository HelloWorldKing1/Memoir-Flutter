# 聊天式记录功能 — 详细设计文档

> **目标**：通过与 AI 机器人自然对话的方式记录灵感、日记、感悟，实现"对话即记录"的交互范式。
>
> **方案**：自定义聊天 UI + LLM API（方案 A），不使用腾讯云 IM UIKit。
>
> **创建日期**：2026-06-25

**核心设计原则：**
- Flutter 层负责 UI + 本地状态，不直接持有 LLM API Key
- 后台服务承担 Prompt 工程、Tool Calling、API Key 安全
- 聊天消息双写：Hive（即时缓存）+ PocketBase（跨设备同步）
- 日记提取为"建议态"，用户确认后才正式创建

## 二、PocketBase 数据模型扩展

在现有 PocketBase 基础上新增两个集合：

### 集合 `conversations`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Text (PK) | 自动生成 |
| `title` | Text | 对话标题，由 AI 从首条消息自动生成 |
| `user` | Relation (users) | 所属用户 |
| `lastMessage` | Text | 最后一条消息摘要（列表展示用） |
| `messageCount` | Number | 消息总数 |
| `created` | DateTime | 自动 |
| `updated` | DateTime | 自动 |

### 集合 `chat_messages`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Text (PK) | 自动生成 |
| `conversation` | Relation (conversations) | 所属对话 |
| `role` | Select | `user` / `assistant` / `system` |
| `content` | Text | 消息正文 |
| `metadata` | JSON | 扩展字段（见下方结构） |
| `user` | Relation (users) | 所属用户 |
| `created` | DateTime | 自动 |

**`metadata` JSON 结构：**
```json
{
  "status": "streaming|complete|error",
  "diaryDraft": {
    "title": "string | null",
    "content": "string | null",
    "entryType": "inspiration|reflection|diary|summary|article|null",
    "mood": "happy|neutral|sad|angry|love|null",
    "weather": "sunny|cloudy|rainy|null",
    "tags": ["string"],
    "confidence": 0.0-1.0,
    "suggestedAction": "create|append_to_last|ask_user"
  },
  "errorMessage": "string | null",
  "tokenUsage": { "prompt": 0, "completion": 0 }
}
```

### 3.3 状态管理 (Riverpod)

```dart
// ---- lib/features/chat/chat_notifier.dart ----

@riverpod
class ChatNotifier extends AsyncNotifier<ChatState> {

  @override
  Future<ChatState> build(String conversationId) async {
    // 1. 先从 Hive 缓存加载消息（即时显示）
    final cached = await _loadFromHive(conversationId);
    // 2. 同时从 PocketBase 拉取最新消息（后台同步）
    _syncFromPocketBase(conversationId);
    return cached;
  }

  // 发送消息
  Future<void> sendMessage(String content) async;

  // 流式接收机器人回复
  void _onStreamChunk(String chunk);
  void _onStreamComplete(DiaryDraft? draft);

  // 取消当前流式回复（页面退出 / 用户手动停止 / 发送新消息中断旧流）
  Future<void> cancelStreaming();

  // 重新生成 AI 回复（删除最后一条 assistant 消息，重新请求）
  Future<void> regenerateResponse();

  // 编辑已发送的用户消息（修改后重新发送，截断后续对话）
  Future<void> editAndResend(String messageId, String newContent);

  // 重试发送
  Future<void> retryMessage(String messageId);

  // 确认保存日记
  Future<void> confirmDiaryDraft(DiaryDraft draft);

  // 删除消息
  Future<void> deleteMessage(String messageId);

  // 释放资源（取消 SSE、关闭 Hive box 监听等）
  @override
  void dispose();
}

// 结构化错误类型（替代 String? error）
enum ChatErrorType {
  network,          // 网络不可用，可自动重试
  authExpired,      // Token 过期，需重新登录
  llmRateLimited,   // LLM 限流，需等待后重试
  llmContentBlocked,// LLM 内容安全拦截，不可重试
  llmTimeout,       // LLM 响应超时
  serverError,      // 后台服务异常
  unknown,
}

class ChatError {
  final ChatErrorType type;
  final String userMessage;       // 用户可读文案
  final bool isRetryable;
  final int? retryAfterSeconds;
}

// ChatState
class ChatState {
  final List<ChatMessage> messages;
  final bool isStreaming;             // 机器人正在生成回复
  final String? streamingText;        // 当前流式文本
  final String? streamingMessageId;   // 🆕 当前流式消息的 ID（用于取消/重试定位）
  final ChatError? error;             // 🆕 结构化错误
  final bool isLoadingMore;           // 加载更早消息
  final bool hasMore;                 // 是否有更多历史消息
  final bool isOffline;               // 🆕 是否处于离线模式
}
```

### 3.4 核心数据流

```
┌─────────────────────────────────────────────────────────────┐
│                    发送消息完整流程                            │
└─────────────────────────────────────────────────────────────┘

用户输入 "今天看到一只流浪猫，它好可怜，我想帮它"

1. UI: chat_input_bar.dart
   ├── onSubmitted(text)
   └── ChatNotifier.sendMessage(text)

2. ChatNotifier.sendMessage()
   ├── 创建 user ChatMessage (status: sending)
   ├── 写入 Hive (乐观更新)
   ├── 更新 state.messages
   ├── 设置 state.isStreaming = true
   ├── 创建 assistant ChatMessage (status: streaming, content: "")
   ├── 调用 LLMService.sendMessage(conversationId, messages)
   └── 返回

3. LLMService (HTTP SSE)
   ├── POST /api/chat
   │   body: { messages: [...history, newMessage], conversationId }
   │   headers: { Accept: text/event-stream }
   ├── 建立 SSE 连接
   └── 逐块接收数据

4. SSE 事件流处理:
   ┌──────────────────────────────────────┐
   │ event: text                          │
   │ data: {"delta": "听起来你遇到了"}      │
   │ → ChatNotifier._onStreamChunk()       │
   │   └── 追加到 streamingText            │
   │                                      │
   │ event: text                          │
   │ data: {"delta": "一只特别的猫咪..."}   │
   │ → 继续追加                            │
   │                                      │
   │ event: tool_call                     │
   │ data: {                              │
   │   "name": "extract_diary",           │
   │   "arguments": {                     │
   │     "title": "偶遇流浪猫",            │
   │     "content": "今天在路上...",       │
   │     "entryType": "reflection",       │
   │     "mood": "sad",                   │
   │     "tags": ["流浪动物","感悟"],       │
   │     "confidence": 0.85               │
   │   }                                  │
   │ }                                    │
   │ → ChatNotifier._onStreamComplete()    │
   │   └── 创建 DiaryDraft 挂到消息上       │
   │                                      │
   │ event: done                          │
   │ data: {"usage": {...}}               │
   │ → 标记消息 status: complete           │
   │ └── 异步写入 PocketBase               │
   └──────────────────────────────────────┘

5. UI 响应:
   ├── message_bubble.dart 实时显示流式文本
   ├── diary_draft_card.dart 弹出"是否保存为日记？"
   └── 用户点击确认 → ChatNotifier.confirmDiaryDraft()
       ├── 调用 DiaryNotifier.createDiary(draft)
       ├── 日记写入 PocketBase + Hive
       └── 显示 SnackBar "日记已保存 ✅"

### 3.4b 消息持久化职责分工

```
消息数据的写入路径（两步，避免 Worker 直接写 PB）：

路径 A: Flutter 客户端 → PocketBase (推荐)
  done event 后:
    ├── user 消息 → 客户端写入 PB (pb.collection('chat_messages').create())
    └── assistant 消息 → 客户端写入 PB

路径 B (备选): Worker → PocketBase
  仅当需要"后台也能看到完整消息"时使用
  风险: Worker 需要 PB Admin Token，增加攻击面

当前推荐路径 A（客户端负责持久化），优势：
- 复用现有 PB 客户端认证
- 不暴露 Admin Token 给 Worker
- Hive → PB 同步逻辑与现有 Diary 模式一致
```

## 四、LLM 集成设计

### 4.1 方案选择：后台代理模式

```
Flutter ──SSE──► 你的后台 ──API──► LLM Provider
                      │
                      ├── 管理 API Key（安全）
                      ├── 注入 System Prompt
                      ├── 管理对话历史裁剪
                      ├── 处理 Function Calling
                      └── 解析结构化输出
```

**为什么不用 Flutter 直接调 LLM API？**
1. API Key 不应打包在客户端
2. System Prompt 需要 A/B 测试和热更新（不依赖发版）
3. Function Calling 结果需要服务端验证
4. 未来可能需要接入多个 LLM Provider 做 fallback

**后台技术选型建议：**

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **Cloudflare Workers** | 轻量、全球部署 | 免运维、低延迟、免费额度大 | 无状态，需外部存储 |
| **Node.js (Express/Fastify)** | 灵活、生态好 | SSE 支持成熟，部署简单 | 需服务器 |
| **Go 服务** | 高性能 | 部署为单文件，低资源占用 | 开发速度较慢 |
| **PocketBase Hook** | 极简（与现有后端复用） | 零新增服务 | PB 的 JS Hook 能力有限 |

**推荐：PocketBase JS Hook + Cloudflare Workers 组合。** MVP 阶段可直接在 PocketBase 的 `pb_hooks` 目录写 JS 逻辑，无需新增服务。

### 4.2 LLM Provider 选择

| Provider | 模型推荐 | 中文能力 | 流式输出 | Function Calling | 价格 (每百万token) |

```

### 4.4 对话历史管理策略

LLM 的上下文窗口有限，需要裁剪策略：

```
┌─────────────────────────────────────────┐
│          LLM 上下文窗口 (32K tokens)       │
│  ┌───────────────────────────────────┐  │
│  │ System Prompt        (~1.5K)       │  │
│  ├───────────────────────────────────┤  │
│  │ 最近 20 条消息        (~variable)   │  │
│  │ (优先保留含 diaryDraft 的消息)      │  │
│  ├───────────────────────────────────┤  │
│  │ 摘要：更早的对话       (~500)       │  │
│  │ (由 AI 定期生成)                    │  │
│  ├───────────────────────────────────┤  │
│  │ Current User Message   (~variable) │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**裁剪规则：**
1. System Prompt 固定占据 ~1,500 tokens
2. 最近 20 条消息完整保留
3. 当对话超过 20 条，触发后台异步生成「对话摘要」，替换早期消息
4. 包含 `diaryDraft` 的消息（已保存的日记）优先保留上下文
5. 用户最近 3 条日记的标题+摘要作为额外上下文注入

### 5.2 Layer 1: 角色系统提示词

```markdown
你是 Memoir，一位温暖、敏锐且富有洞察力的个人记录助手。
你的使命是通过自然对话，帮助用户捕捉灵感碎片、记录日常感悟、

## 你的性格
- 温暖而有共情力：你理解用户的情绪，回应有温度
- 简洁而聚焦：回复通常 2-4 句话，不啰嗦，直击要点
- 善于追问：当信息不足以构成完整记录时，温和地追问补充
- 尊重界限：用户不想记录时，不强行引导

## 你的核心能力
你可以从对话中识别并提取以下 6 种内容类型：

| 类型 | 特征 | 示例场景 |
|------|------|----------|
| inspiration | 一闪而过的想法、创意、金句 |"突然想到一个好点子..."|
| reflection | 对事件的感受、思考、省思 |"今天这件事让我意识到..."|
| diary | 对一天/事件的完整记录 |"今天发生了这些事..."|
| summary | 学习笔记、读书/观影总结 |"刚看完这本书，核心观点是..."|
| article | 需要展开的长文草稿 |"我想写一篇关于..."|

## 你的行为准则
1. 用户消息明显是日记/灵感内容 → 自然回应 + 触发 extract_diary
2. 用户消息是闲聊/问候/问题 → 自然回应，不强行提取
3. 用户说"记录/记一下/帮我整理" → 明确触发提取

5. 心情和标签不确定时，设为 null 并询问用户

### 5.5 Prompt 决策树

```

    └── 情绪倾诉（非记录意图）

### 6.4 Provider Adapter 接口

```typescript
// 统一 LLM Provider 接口
interface LLMProvider {
  name: string;
  models: string[];

  // 流式对话
  chatStream(
    messages: Message[],
    options: ChatOptions
  ): AsyncGenerator<SSEEvent>;

  // 非流式对话（用于摘要生成等后台任务）
  chat(
    messages: Message[],
    options: ChatOptions
  ): Promise<ChatResponse>;
}

interface ChatOptions {
  model: string;
  temperature?: number;     // 默认 0.7
  maxTokens?: number;       // 默认 2048
  functions?: FunctionDef[]; // Function Calling 定义
  functionCall?: 'auto' | 'none' | { name: string };
}

type SSEEvent =
  | { type: 'text'; delta: string }
  | { type: 'tool_call'; name: string; arguments: object }
  | { type: 'done'; usage: TokenUsage }
  | { type: 'error'; message: string };
```

```

- 免费额度: 每天 10 万次请求
- 全球边缘节点，低延迟（尤其对中国用户有亚太节点）
- 内置 KV 存储（可缓存 Prompt 模板）
- 支持 WebSocket / SSE（通过 Streams API）
- 一行命令部署: npx wrangler deploy

限制:
- 单次请求 CPU 时间 30s（SSE 流式场景需注意超时）
- 无状态（对话历史存 PocketBase）
```

## 九、关键技术决策与权衡

| 决策点 | 推荐选项 | 备选 | 理由 |
|--------|---------|------|------|

| 对话管理 | 多对话模型 | 单对话 | 允许用户按主题分离对话 |
| 日记提取 | 自动 + 用户确认（置信度阈值） | 纯手动触发 | 兼顾效率与质量 |

## 十、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM API 不稳定/涨价 | 服务中断 | Provider Adapter 模式，支持一键切换 + Fallback |
| SSE 长连接在移动端不稳定 | 消息丢失 | 断线重连 + 消息 ID 去重 + 本地缓存 |
| LLM 幻觉（编造日记内容） | 数据质量 | confidence 阈值 + 用户确认机制 |
| Token 消耗过大 | 成本上升 | 历史裁剪 + 摘要压缩 + 限制单次上下文长度 |
| 用户不想让 AI 知道太多 | 隐私顾虑 | 所有数据存于用户自托管 PocketBase，Prompt 中可控制信息暴露程度 |
| **离线时无法使用 AI** | 核心功能不可用 | 见第十一章离线策略 |
| **Cloudflare Workers 30s 超时** | 长回复中断 | Workers 逐块转发（不等待完整 LLM 响应）；30s 内 LLM 未完成则发送 `event: timeout`，客户端提示"回复较长，请稍候重新生成" |
| **多设备同时打开聊天页** | 消息重复/SSE 冲突 | 见第十五章多设备同步 |

## 十一、离线策略与 SSE 生命周期

> 项目核心原则是「离线优先」。但 AI 聊天本质依赖网络——本章定义离线时的降级行为。

### 11.1 三级可用性模型

```
┌─────────────────────────────────────────────────────────┐
│  L1: 完全在线（网络可用 + 后台可达 + LLM 可达）            │
│  → 完整聊天体验：发送消息、流式回复、日记提取              │
├─────────────────────────────────────────────────────────┤
│  L2: 弱网 / 后台不可达（网络可用但 API 超时/5xx）         │
│  → 消息可输入，发送时标记 pending                         │
│  → 显示 "正在尝试重新连接..." 横幅                        │
│  → 自动重试（指数退避: 1s→2s→4s→8s→max 60s）             │
├─────────────────────────────────────────────────────────┤
│  L3: 完全离线（无网络）                                   │
│  → 可查看历史消息（Hive 缓存）                            │
│  → 可输入消息草稿（本地暂存，恢复网络后发送）               │
│  → 显示离线横幅 + "联网后可发送消息"                      │
│  → 对话列表正常可浏览                                     │
└─────────────────────────────────────────────────────────┘
```

### 11.2 离线消息队列

```dart
// ChatState 中新增字段
class ChatState {
  // ... existing fields ...
  final bool isOffline;                          // 当前是否离线
  final List<PendingMessage> pendingMessages;    // 等待发送的消息队列
}

class PendingMessage {
  final String tempId;           // 临时 ID（UUID），发送成功后替换为服务端 ID
  final String conversationId;
  final String content;
  final DateTime createdAt;
  int retryCount;
  DateTime? nextRetryAt;         // 指数退避后的下次重试时间
}
```

**离线发送流程：**
```
用户输入消息 (L3 离线)
  ├── 创建 PendingMessage (tempId = uuid)
  ├── 创建 user ChatMessage (status: pending, id = tempId)
  ├── 写入 Hive
  ├── UI 显示为 "发送中" 状态（灰色指示器）
  └── 监听 connectivity_plus 网络恢复事件

网络恢复 (L3 → L1)
  ├── connectivity_plus 通知
  ├── 遍历 pendingMessages (FIFO)
  ├── 逐个调用 sendMessage()
  ├── 成功 → 替换 tempId 为服务端 messageId → 移除 PendingMessage
  └── 失败 → retryCount++ → 指数退避
```

### 11.3 SSE 连接生命周期

```
SSE 连接状态机:
┌──────┐   sendMessage()   ┌───────────┐
│ idle │ ─────────────────► │ connecting │
└──────┘                   └─────┬─────┘
      ▲                         │ HTTP 200 + SSE stream
      │                         ▼
      │                    ┌───────────┐
      │                    │ streaming │ ← _onStreamChunk() 持续更新 UI
      │                    └─────┬─────┘
      │                         │
      │              ┌──────────┼──────────┐
      │              │          │          │
      │         done event  error event  timeout/网络断开
      │              │          │          │
      │              ▼          ▼          ▼
      │         ┌────────────────────────────┐
      │         │        cleanup              │
      │         │ - 标记 assistant 消息状态    │
      │         │ - 写入 PocketBase           │
      │         │ - 释放 SSE 连接              │
      │         └────────────┬───────────────┘
      │                      │
      └──────────────────────┘
```

**关键取消场景：**

| 场景 | 行为 |
|------|------|
| 用户点击「停止生成」按钮 | `cancelStreaming()` → 关闭 SSE → assistant 消息保留已接收的部分 (status: complete) |
| 用户在流式过程中发送新消息 | 自动取消旧 SSE → 保留已接收部分 → 发送新消息 |
| 页面 `dispose()` (退出聊天页) | `cancelStreaming()` → 关闭 SSE → assistant 消息保留已接收部分 → 持久化 |
| App 进入后台 | **不取消** SSE（保持连接，后台继续接收）。iOS 注意后台任务时限 |
| SSE 连接意外断开 | 自动重连 1 次（携带最后收到的 messageId，服务端从中断点恢复）。重连失败 → 标记 error |

### 11.4 SSE 中断恢复协议

```
POST /chat/resume
Headers: Authorization: Bearer {token}

Request:
{
  "conversationId": "abc123",
  "lastReceivedMessageId": "msg_xyz",   // 客户端最后收到的消息 ID
  "lastChunkIndex": 42                  // 服务端在 SSE 中附带递增序号
}

Response (SSE): 从第 43 个 chunk 开始续传
```

## 十二、错误处理体系

### 12.1 分层错误映射

```
┌─────────────────────────────────────────────────────────┐
│                     UI 层 (用户可见)                       │
│  "网络连接失败，请检查网络设置"                             │
│  "AI 服务暂时繁忙，3 秒后自动重试"                          │
│  "登录已过期，请重新登录"                                  │
└───────────────────┬─────────────────────────────────────┘
                    │ ChatError.userMessage
┌───────────────────▼─────────────────────────────────────┐
│               ChatNotifier (错误转换)                      │
│  DioException → ChatError → state.error                  │
│  SSE error event → ChatError                             │
│  timeout → ChatError.llmTimeout                          │
└───────────────────┬─────────────────────────────────────┘
                    │
┌───────────────────▼─────────────────────────────────────┐
│             LLMService / 后台 (原始错误)                   │
│  网络: SocketException, TimeoutException                 │
│  认证: 401 Unauthorized                                  │
│  LLM:  429 Rate Limit, 400 Content Blocked, 500 Server   │
│  SSE:  connection lost, chunk parse error                │
└─────────────────────────────────────────────────────────┘
```

### 12.2 错误码映射表

| 原始错误 | ChatErrorType | UI 文案 | 重试策略 |
|----------|--------------|---------|----------|
| `SocketException` | `network` | "网络连接失败，请检查网络" | 自动重试 3 次 |
| `TimeoutException` | `network` | "连接超时，请稍后再试" | 自动重试 2 次 |
| HTTP 401 | `authExpired` | "登录已过期，请重新登录" | 跳转登录页 |
| HTTP 429 | `llmRateLimited` | "AI 服务繁忙，{n}秒后自动重试" | 等待 Retry-After |
| HTTP 400 (content_filter) | `llmContentBlocked` | "该内容无法处理，请修改后重试" | 不可重试 |
| HTTP 500 | `serverError` | "服务异常，请稍后再试" | 手动重试 |
| SSE 断连 | `network` | "连接中断，正在重新连接..." | 自动重连 1 次 |
| Cloudflare 524 | `llmTimeout` | "回复时间较长，请稍候重新生成" | 手动重试 |

### 12.3 异常恢复流程

```
流式回复中发生错误:
  ├── state.error = ChatError(...)
  ├── assistant 消息 status → error
  ├── assistant 消息 errorMessage = ChatError.userMessage
  ├── 消息气泡底部显示红色错误文案 + [重试] 按钮
  └── 用户: 点击 [重试] → regenerateResponse()
      └── 如果 conversationId 无变化，仅重发最后一条 user 消息的上下文
```

## 十五、多设备同步策略

### 15.1 同步目标

| 数据 | 优先级 | 策略 |
|------|--------|------|
| 消息历史 | P0 | PocketBase 为准，Hive 为缓存 |
| 对话列表 | P0 | PocketBase 排序，Hive 缓存 |
| 进行中的流式消息 | — | 不同步（仅当前设备可见） |
| 草稿消息 (未发送) | P1 | 暂存本地，不同步 |
| diaryDraft (未确认) | P1 | metadata 随消息同步，多设备可见 |

### 15.2 并发场景处理

```
场景: 用户在手机和桌面同时发送消息

手机发送 msg_A → 后台处理中...
桌面发送 msg_B → 后台处理中...

服务端:
  ├── 消息按到达时间排序
  ├── 每个 conversation 加锁（同一对话同时只有一个 LLM 请求在处理）
  └── 后续请求返回 409 Conflict → 客户端合并消息

客户端 (桌面):
  ├── 收到 409 → 提示 "检测到其他设备正在发送消息"
  └── 拉取最新消息 → 发现 msg_A 已存在 → 追加到本地列表
```

### 15.3 跨设备消息去重

```
每条消息携带 idempotency_key (客户端生成的 UUID):
{
  "idempotencyKey": "uuid-v4",    // 发送时生成
  "message": "用户文本"
}

服务端去重:
- Redis / KV 存储 idempotency_key → messageId 映射（TTL 24h）
- 重复请求直接返回已有 messageId
- PocketBase 作为兜底（查询最近 10 条消息的 idempotency_key）
```

### 16.2 数据迁移策略

```
场景 1: 模型新增字段
  例: ChatMessage 新增 isPinned 字段
  → 使用 Hive 的 @HiveField 注解时指定新 index
  → 旧数据读取时该字段为 null → 使用默认值 ?? false

场景 2: typeId 冲突
  → 预留 100+ 区间避免与现有 ID 冲突
  → 检查现有项目 Hive 适配器中的最大 typeId

场景 3: Hive box 迁移到 PocketBase
  → 首次接入时全量同步
  → 增量通过 updated 字段对比
```

### 16.3 Hive ↔ PocketBase 同步时机

```
写入:
  ┌── 用户发送消息
  │   ├── 写入 Hive (即时, status: pending)
  │   └── 发送成功后
  │       ├── 更新 Hive (status: sent, id = 服务端ID)
  │       └── 写入 PocketBase (异步, 静默失败不阻塞 UI)
  │
  └── AI 流式回复 complete
      ├── 更新 Hive (status: complete)
      └── 写入 PocketBase

读取:
  ├── ChatNotifier.build()
  │   ├── 立即返回 Hive 缓存数据 (UI 即时展示)
  │   ├── 后台拉取 PocketBase 最新数据
  │   └── 合并策略: 以 updated 字段为准 (Last-Write-Wins)
  └── 跨设备同步
      ├── 订阅 PocketBase SSE (如果 PB 支持) 或 定时轮询
      └── 检测新消息 → 追加到 Hive → 更新 state.messages
```

### 17.2 Prompt Injection 防护

```
用户输入的安全处理:
1. 长度限制: 单条消息 max 2000 字符 (前端 + 后端双重校验)
2. 指令注入防护:
   - 用户消息作为 User Role 发送，不拼接到 System Prompt 中
   - System Prompt 与 User Message 严格分离
   - 不在 Prompt 中使用 "忽略上述指令" 可绕过的模式
3. 链接/代码注入: 用户消息中的 URL 和代码不执行，仅作为文本传递
4. 限流: 每用户每分钟 max 10 条消息 (Cloudflare Rate Limiting)
5. 内容安全: 可选接入 LLM 自带的内容安全过滤 (DeepSeek 默认开启)
```

### 17.3 数据隔离保证

```
PocketBase 层面:
  - conversations / chat_messages 集合规则: @request.auth.id = user.id
  - 后台服务使用 Admin Token 只做验证，业务逻辑用 userId 过滤

后台层面:
  - 每个请求从 JWT 提取 userId
  - 所有 DB 查询强制带 userId 条件
  - 不允许跨用户访问对话/消息

LLM 层面:
  - 每次请求仅注入当前用户的最近日记摘要
  - 不同用户的对话完全隔离，无共享上下文
```

### 17.4 CORS 配置

Cloudflare Worker 需要配置:
  - Access-Control-Allow-Origin: * (开发阶段) / 指定域名 (生产)
  - Access-Control-Allow-Headers: Authorization, Content-Type
  - Access-Control-Allow-Methods: POST, GET, DELETE
  - 处理 OPTIONS 预检请求

PocketBase Admin UI → Settings → CORS:
  - 添加 Worker 域名和客户端域名
